## Context

- **B (compositional)**: 1,541 — need multi-row mappings (assessment + qualifier)
- **C (unmappable)**: 38,192 — no clinical attributes detected by regex

A previous sample of 1,000 C items showed **89.6% genuinely non-clinical** (admin, template artifacts, section headers) and **10.4% containing clinical terms** that the regex missed. That suggests ~4,000 items may be rescuable.

## Approach: Item-by-Item Clinical Reasoning

**Do NOT use batch regex/grep approaches.** An earlier attempt (session 1) used regex pattern matching to bulk-assign broad OMOP concepts. This produced many false positives (e.g., "Allergies Reviewed" mapped as an allergy finding, "Did you take a sleep aid?" mapped as a sleep observation). That commit was reverted.

The correct approach:

1. **Read each item** from the enriched CSV with full context (template name, group name, value type, custom list values, units, min/max)
2. **Reason about the clinical meaning** — what would a clinician actually record in this field? Is it a measurement, finding, observation, or administrative artifact?
3. **Look up OMOP concepts** via vocabulary tools (`search_concepts`, `get_concept_descendants`) — search for specific terms, not broad categories
4. **Record the decision** in `clinical_review.csv` with full metadata including qualifiers
5. **Skip genuinely non-clinical items** — most C items are admin/template artifacts

### Decision categories
- **map**: Item has clinical value, mapped to specific OMOP concept(s)
- **skip**: Item is non-clinical (admin, template, section header, duplicate)
- **flag**: Ambiguous — needs source data review to determine clinical value

### Manual batches (for smaller runs)

1. Read enriched CSV starting at the next unreviewed line
2. For each item, reason about clinical value using the template/group/value context
3. Look up OMOP concepts via `search_concepts` for items that appear clinical
4. Append decisions to `clinical_review.csv`
5. Work in batches of ~20-50 items per session

### Tips for reviewers
- Items in the same template/group often share clinical context — process them together
- Custom list values are the strongest signal: `Uninvolved|Mild|Moderate|Severe` = clinical scale
- `val_type=numeric` with units and min/max = likely a measurement
- `val_type=category` or `custom_list` with clinical terms = likely an observation
- Items appearing in >3 templates are more likely clinical (widely used)
- Section headers, "Comments", "Other", free-text fields are usually non-clinical

## Mapping Philosophy — MAP AGGRESSIVELY

**Find EVERY possible OMOP concept match.** When in doubt, map it.

Prior sessions (1-14) used a conservative approach, skipping SDOH screening,
nursing ROS, medical history screening questions, behavioral observations, and
violence risk assessments as "non-clinical." This was WRONG. Starting session 15,
map anything that captures health information and has an OMOP concept.

Only skip truly non-clinical items: staff names, consent checkboxes, belongings
inventory, equipment IDs, free-text comment fields, section headers.

## What NOT to Do

- Don't use batch regex/grep to assign concepts — reason about each item individually
- Don't assign broad OMOP concepts without checking if a more specific one exists
- Don't skip items just because they're "screening questions" — if they capture health data, map them
- Don't modify pipeline code until the review CSV is complete
- Don't skip items without recording the skip decision in the CSV
